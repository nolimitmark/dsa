##  [55. Jump Game](https://leetcode.com/problems/jump-game/)

**Key Observation**:

1. If you encounter an index before the last index with a maximum jump length of `0`, then you can never reach the last index.

*Now let's dive into solving and understanding this problem with 2 different approaches, from the best to the least optimal.*

## Approach 1: Greedy

**Intuition:**

To ensure we reach the end, we aim to maximize our jump length at each index. We start by initializing a variable `jump` to the first item in the `nums` array (nums[0]). We then iterate from the second item to the last item. In each iteration, we decrease `jump` by 1, representing the step taken at the current index. If the maximum jump length at the current index is greater than our `jump`, we update `jump` to this new value, following the greedy approach.

And if our jump ever comes down to `0` and we're not at the last index we want to simply `return false` because we  can't go any further.

**Let's take a look at this fun example to understand this intuition:**

Imagine you're playing a runner game where you start with just one runner. As you run along the track, you encounter power-ups that can increase the number of your runners. These power-ups come in different forms, like multiplying your runners by a certain number or adding a fixed number of runners.

As you can see in the video above, we initially start with one runner and then we came across two power-ups, which are:

1. One that adds 80 runners (+80)
2. One that multiplies your runners by 3 (*3)

To maximize our chances of progressing further and defeating enemies, we must choose wisely between these options. Choosing the first power-up (+80) increases our runners to `(1 + 80)`, allowing us to defeat the 45 enemies ahead. This decision, where we grab the power-up that maximizes our number of runners, exemplifies a greedy strategy.

What happens if we don't choose greedily?

As shown, opting for the second power-up (multiplying by 3) only increases our runners to `(1 * 3)`, which isn't sufficient to defeat the 42 enemies ahead.

***Note: If we ever run out of players before we reach the end of the game, we lost the game.***



**Algorithm:**

1. Initialize a variable `n` to the length of the `nums` array.
2. Initialize a variable `jump` to the jump length at the `0th` index.
3. Add a conditional check to `return false` if `jump` is `0` and `n > 1`, This handles cases where the initial jump is `0`, preventing further progress if more jumps are required.
4. Iterate from index `1` to `n-1`:
   - Subtract `1` from `jump`: `jump -= 1`.
   - Update `jump` to the maximum value between `jump` and the jump length at the current index
   - If `jump` becomes `0` and `i < (n - 1)`, `return false` because we can't proceed further if unable to jump.
5. `return true` at the end because reaching the last index is possible.

**Implementation:**

```python
class Solution:
    def canJump(self, nums: List[int]) -> bool:
        n = len(nums)
        jump = nums[0]
        if jump == 0 and n > 1:
            return False
        
        for i in range(1, n):
            jump -= 1
            jump = max(jump, nums[i])
            
            if jump == 0 and i < (n - 1):
                return False
        
        return True
```

**Complexity Analysis**

- **Time complexity:** The algorithm iterates through the `nums` array once, which takes O(n) time. Within each iteration, it performs constant-time operations such as comparisons and updates. Therefore, the time complexity is dominated by the linear traversal of the array, resulting in **O(n)**.
- **Space complexity: **The algorithm uses a constant amount of extra space, independent of the size of the input array. Specifically, it maintains a few variables (`n`, `jump`, and `i`) that do not scale with the size of `nums`. Hence, the space complexity is **O(1)**.



## Approach 2: Dynamic Programming

**Intuition:**

This approach involves using a dynamic programming bottom-up method. We start by creating a `dp` array where each element represents whether it's possible to reach the end of the array from that index. Starting from the end, we iterate backward, marking positions that can reach the end directly or via other reachable positions.

For each index, we calculate the maximum index we can reach as `max_index = index + nums[index]`. We then take the minimum between the `max_index` and the last index `n - 1` to ensure we don't go out of bounds in the array.

We iterate from the next index `index + 1` to the maximum index we can reach `max_index + 1`. In each iteration, we check if we can reach the end from that index (`if dp[j] is true`). If it is, we set `dp` at our current index to `true` and break out of the loop, as we've found a path to the end.

We continue this process until we reach the first index `0`, and finally return whether `dp[0]` is `true` or `false`.

**To Illustrate This Intuition, Let's Examine an Example:**

let `n` be the length of our nums array.

Using our example array, `nums = [2, 3, 1, 1, 4]`, we initialize a boolean array `dp` of size `n`, with all values set to `False` except for the last item, which is set to `True`. This is because the last item in the array can always reach itself. If our array had only one item, it would be the last item and could trivially reach the end on its own.

Here's the initial state:

- `dp = [False, False, False, False, True]`

Now, we iterate from the second to last index to the first index`(n - 2)` - `0`

- **Index 3:**
- `max_index = min(3 + nums[3], 4) = min(3 + 1, 4) = 4`

- Iterate through all indexes we can reach (index + 1) -> (max_index + 1) = 4 -> 5
  - `dp[4]` is `True`, so we set `dp[index] = True` and break out of the loop.

- Updated `dp`: `[False, False, False, True, True]`

* **Index 2:**

  - `max_index = min(2 + nums[2], 4) = min(2 + 1, 4) = 3`

  - Iterate through all indexes we can reach (index + 1) -> (max_index + 1) = 3 -> 4
    - `dp[3]` is `True`, so we set `dp[index] = True` and break out of the loop.

  - Updated `dp`: `[False, False, True, True, True]`



* **Index 1:**

  - `max_index = min(1 + nums[1], 4) = min(1 + 3, 4) = 4`

  - Iterate through all indexes we can reach (index + 1) -> (max_index + 1) = 2 -> 5
    - `dp[2]` is `True`, so we set `dp[index] = True` and break out of the loop.

  - Updated `dp`: `[False, True, True, True, True]`



* **Index 0:**

  - `max_index = min(0 + nums[0], 4) = min(0 + 2, 4) = 2`

  - Iterate through all indexes we can reach (index + 1) -> (max_index + 1) = 1 -> 3
    - `dp[1]` is `True`, so we set `dp[index] = True` and break out of the loop.

  - Updated `dp`: `[True, True, True, True, True]`

**Algorithm:**

1. Initialize a variable `n` to the length of the `nums` array.
2. Initialize a boolean array `dp` of size `n` to all `false`.
3. Set the last item in the `dp` array to `true` `dp[n - 1] = true`.
4. Iterate `nums` backwards from the second to last index to the start index `nums[n - 2] - 0`.
   - For each iteration `i`, calculate the maximum index `max_index` we can reach from there, ensure it doesn't go out of bounds, it should at most be `n - 1`.
   - Iterate from the next index `i + 1`to our maximum index `max_index` 
     - For each iteration `j` ,  if `dp[j] == True` update `dp[i] to `true` as well and break out of the loop.
5. Finally return `dp[0]`.

**Implementation:**

```python
class Solution:
    def canJump(self, nums: List[int]) -> bool:
        n = len(nums)
        dp = [False] * n
        dp[n-1] = True
        
        for i in range(n-2, -1, -1):
            furthest_jump = min(i + nums[i], n - 1)
            for j in range(i + 1, furthest_jump + 1):
                if dp[j]:
                    dp[i] = True
                    break
        
        return dp[0]
```



**Complexity Analysis**

- **Time complexity:** The algorithm uses a nested loop where the outer loop iterates through the `nums` array from the second-to-last element to the first element, resulting in **O(n)** iterations. The inner loop, in the worst case, iterates through almost all elements ahead of the current index, leading to **O(n)** iterations for each element of the outer loop. Therefore, the overall time complexity is **O(n^2)**.
- **Space complexity:** The algorithm uses a dynamic programming array `dp` of the same length as the input array `nums`, which stores boolean values indicating whether each index can reach the end. This results in a space complexity of **O(n)** due to the storage requirement for the `dp` array.
