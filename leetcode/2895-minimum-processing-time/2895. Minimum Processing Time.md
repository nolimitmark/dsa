## [2895. Minimum Processing Time](https://leetcode.com/problems/minimum-processing-time/)

> *Before you continue with this article, ensure you've tried this problem on your own at least.*

### Introduction

This is a greedy problem so let's start by understanding the greedy intuition behind it.

We are provided with two integer arrays `processorTime` and `tasks` where `processorTime` represents the time each processor becomes available and `tasks` represents how long each task takes to complete. Since we are trying to find the minimum time needed to complete all tasks, we should process tasks that takes longer time to complete first. 

Let's break down why we should process longer tasks first in this scenario:

1. **Limited Resources**: Each processor has 4 cores, which means it can handle 4 tasks simultaneously.
2. **Goa**l: We want to minimize the overall completion time, which is determined by the processor that finishes last.
3. **Greedy Strategy**: By assigning longer tasks to processors that become available earlier, we're making efficient use of the earlier start times.

Here's why this strategy works:

1. If we put shorter tasks on processors that become available early, we might end up with long tasks on processors that become available late. This could lead to these processors finishing much later than the others, increasing our overall completion time.
2. By putting longer tasks on processors that become available early, we're "front-loading" the work. This increases the chance that all processors will finish at roughly the same time, minimizing the maximum completion time.
3. Shorter tasks are more flexible. They can be used to "fill in the gaps" and can be completed quickly even if they start later.

### **Implementation**

- First we start by sorting the both arrays, we will sort the `processorTime` in ascending order while the `tasks` in descending order since we want to prioritise longer tasks first.

  ```python
  processorTime.sort()
  tasks.sort(reverse=True)
  ```

- Now we can use two pointers to iterate through the both arrays:

  - The `task_pointer` moves through the `tasks` array, incrementing after each iteration. This represents assigning each task in order.
  - The `processor_pointer` moves through the `processorTime` array, incrementing only after every 4 iterations. This represents moving to the next processor after its 4 cores have been used.

  Our `answer` is the maximum `processorTime[processor_pointer] + tasks[task_pointer]` we will ever find. 

  ```python
  answer = 0
  processor_pointer = 0
  core = 1 # this variable will track the number of cores we've used for each processor
  for task_pointer in range(len(tasks)):
      if core > 4:
          # we've used the 4 cores for the current processor, time to switch to a new processor and reset the core
          core = 1
          processor_pointer += 1
          
    	answer = max(answer, processorTime[processor_pointer] + tasks[task_pointer])
      core += 1   
  ```

**Putting it all together:**

> ```python
> class Solution:
>     def minProcessingTime(self, processorTime: List[int], tasks: List[int]) -> int:
>         processorTime.sort()
> 		tasks.sort(reverse=True)
>         
>         answer = 0
>         processor_pointer = 0
>         core = 1
>         for task_pointer in range(len(tasks)):
>             if core > 4:
>                 core = 1
>                 processor_pointer += 1
> 
>             answer = max(answer, processorTime[processor_pointer] + tasks[task_pointer])
>             core += 1   
>         
>         return answer
> ```

The **Time Complexity** of the program is `O(nlogn)` due to the sorting, where `n` is the length of the tasks, while the **Space Complexity** remains `O(1)` because no significant extra memory was used.

### **Further Optimization**

After sorting the tasks array, we can make a key observation: since each processor has 4 cores, we only need to consider the most time-consuming task in each segment (dividing the `tasks` array into groups of 4) for each processor. This allows us to significantly reduce the number of comparisons we need to make.

- Here, we will sort the `processorTime` in descending order while the `tasks` in ascending order.

  ```python
  processorTime.sort(reverse=True)
  tasks.sort()
  ```

- Now we can iterate through the tasks array much more efficiently:

  - We only look at every 4th element in each segment, which represents the most time-consuming task for each processor.
  - We pair these with processors in order, as we've sorted `processorTime` to have the latest available processors first.

  ```python
  answer = 0
  processor_pointer = 0
  
  for task_pointer in range(3, len(tasks), 4): # we start iterating from 3 because it's a 0-indexed array so the last number in the first segment is at index 3 and we also add 4 to the task pointer after each iteration since we only care about the largest number in each segmen t
      answer = max(answer, processorTime[processor_pointer] + tasks[task_pointer])
      processor_pointer += 1 # we always increment this pointer since we're only checking the last number in each segment
  ```

**Putting it all together:**

> ```python
> class Solution:
>     def minProcessingTime(self, processorTime: List[int], tasks: List[int]) -> int:
>         processorTime.sort(reverse=True)
>         tasks.sort()
> 
>         ans = 0
>         processor_pointer = 0
>         
>         for i in range(3, len(tasks), 4):
>             ans = max(ans, tasks[i] + processorTime[core])
>             processor_pointer += 1
> 
>         return ans
>         
> ```

Here, while the **Time Complexity** and **Space Complexity** remains the same as the previous, This optimization reduces our iteration from O(n) to O(n/4), which, while still linear in complexity, provides a significant constant factor improvement in performance.
