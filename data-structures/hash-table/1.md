## **Definition of Hash Tables**

A hash table is a data structure that uses an array to store key-value pairs, allowing for efficient insertion, lookup, and removal operations. It works by using a hash function to map each key to a unique index in the array. The key acts as the identifier for the location, while the value holds the actual data to be stored.

### **What is Hashing?**

Hashing is the process of converting a given key or string of characters into another value, typically a shorter, fixed-length representation. This transformed value serves as a unique identifier, making it easier to locate or reference the original key.

Our hash function can be designed in many ways. For instance, we could create a simple hash function that takes in a key (string) and works like this:

1. Take the ASCII value of each character in the key.
2. Add up all these values.
3. Use the modulo operator with the size of the hash table to get the index.

Example:
Suppose our hash table has a size of 10, and the key is "cat".

1. ASCII values: `'c' = 99`, `'a' = 97` `'t' = 116`
2. Sum of ASCII values: = `99 + 97 + 116 = 312`
3. Compute index: `312 mod  10 = 2`

So, the hash function maps the key "cat" to index 2, therefore the data will be stored at index 2 of the array used in our hash table.

This is just one example, and hash functions can be made more complex depending on the requirements.

**Collisions in Hashing**

Yes, depending on how our hash function is designed, it is possible for different keys to map to the same index. This scenario is called a collision, which occurs when two or more keys point to the same array index.

Collisions can be resolved using techniques such as:

1. **Chaining**
2. **Open Addressing**
3. **Rehashing**

**Chaining**: In this technique, the hash table's array is initialized as an array of data structures, such as linked lists or arrays. When data is hashed to a particular index, it is added to the data structure at that index.

**Open Addressing**: This is a collision resolution technique where all elements are stored directly in the hash table itself, rather than using separate data structures like linked lists. When a collision occurs, the hash table probes for the next available slot in the array using specific probing methods. Here’s how the common methods work:

1. **Linear Probing**:
   - After a collision, the table checks the next slot sequentially (e.g., index `i+1`, `i+2`,…) until an empty slot is found.
   - This method is simple but can cause clustering, where consecutive slots fill up, leading to performance degradation.
2. **Quadratic Probing**:
   - Instead of checking slots sequentially, this method checks slots at intervals that grow quadratically (e.g., `i+1^2`, `i+2^2`,…).
   - This reduces clustering compared to linear probing, but care must be taken to ensure all slots are accessible.
3. **Double Hashing**:
   - A secondary hash function is used to calculate the step size for probing. For example, if the first hash gives index `i`, the next index is calculated as `i + j * h2(key)`, where `j` is the probe count and `h2(key)` is the secondary hash.
   - This method minimizes clustering and improves performance by spreading keys more uniformly.

**Rehashing**: this is a technique used to maintain the efficiency of a hash table as its load factor increases. A high load factor indicates that the table is becoming crowded, which can lead to frequent collisions and slower operations. Rehashing resolves this issue by creating a larger hash table and redistributing the elements into the new table using a new hash function or the same hash function applied to the larger table size.

**What is Load Factor** 

The load factor of a hash table is a measure of how full the table is. It is defined as the ratio of the number of elements (keys) stored in the hash table to the total number of available slots (buckets) in the table.

Load Factor = `Number of Elements / Total Number of Slots`

For example, if a hash table has 100 slots and 60 elements are stored in it, the load factor is `0.60` or `60%`.

A high load factor can increase the likelihood of collisions, while a low load factor can lead to wasted space. To balance these concerns, many hash table implementations dynamically resize the table when the load factor exceeds a certain threshold.

## **Hash Table Operations**

There are several operations that can be performed on a hash table, but here are the ba  sic ones:

- Insertion
- Look-Up
- Retrieval
- Removal

### **Insertion**

This operation involves adding a key-value pair into the hash table. A hash function is used to compute the index, and the key-value pair is stored at that index. If a collision occurs, any of the collision resolution techniques are used to find the next available slot.

The **Time Complexity** in the best case is `O(1)`, but in the worst case, if a collision occurs, it can grow up to `O(n)`.

For example, if all keys map to a particular index, and the collision resolution technique being used is:

1. **Chaining:** In the worst case, all `n` keys are stored in the same chain, and insertion would require traversing the chain, resulting in `O(n)` time complexity.
2. **Probing**: In the worst case, if the hash table is almost full, multiple probes may be required to find an empty slot, leading to `O(n)` time complexity.
3. **Rehashing**: If the hash table’s load factor exceeds a certain threshold (e.g., 0.7), the table is resized, and all keys are reinserted which will take `O(n)`time.

The **Space Complexity** is `O(1)`, as only the key-value pair is added during insertion. However, additional space may be required for:

- **Rehashing**: For allocating a larger hash table when resizing.

------

### **Look-Up**

This operation checks if a specific key exists in the hash table. The hash function computes the index, and the key at that index is compared to the requested key. If they match, the search is successful.

The **Time Complexity** is O(1) on average, but can degrade to O(n) in the worst case when many collisions occur. The **Space Complexity** is O(1), as no additional space is required beyond the current table structure.

------

### **Retrieval**

This operation retrieves the value associated with a given key. After the key’s index is calculated using the hash function, the corresponding value is returned from the hash table.

The **Time Complexity** is O(1) on average, assuming minimal collisions. The **Space Complexity** is O(1), since the retrieval does not require extra space.

------

### **Removal**

This operation involves deleting a key-value pair from the hash table. The key is hashed to find its index, and the pair is removed from that position. If the table uses open addressing, care must be taken to maintain the integrity of the probing sequence.

The **Time Complexity** is `O(1)` on average, but in the worst case, it can degrade to `O(n)` due to collisions.

For example, if all keys map to a particular index, and the collision resolution technique being used is:

1. **Chaining**: In the worst case, all `n` keys are stored in the same chain, and removal would require traversing the chain, resulting in `O(n)` time complexity.
2. **Open Addressing**: After hashing, the table must probe to locate the key based on the probing sequence used during insertion. In the worst case, particularly if the table is densely populated or clustering occurs, locating the key might require up to `O(n)` probes. Additionally, after removing the key, an indicator is placed to maintain the integrity of the probing sequence. This ensures that subsequent searches for other keys in the same sequence are not disrupted.

The **Space Complexity** is `O(1)`, as no extra space is needed beyond the existing table.

## **Applications of Hash Tables**

1. Hash tables are used in databases for indexing, allowing fast retrieval of records based on a unique key such as a user ID or product number.
2. Caching systems use hash tables to store frequently accessed data in memory, providing quick access and reducing database load.
3. Symbol tables in compilers utilize hash tables to store variables, functions, and other identifiers, enabling fast lookups during code compilation.
4. Hash tables are used in storing and managing sessions in web applications to associate session IDs with user-specific data, allowing for quick access to session information.

## **Advantages of Hash Tables**

1. Hash tables offer constant-time average complexity (O(1)) for insertions, deletions, and lookups, making them highly efficient for tasks requiring fast access to data.
2. With the right hash function and load factor, hash tables make efficient use of memory while providing quick access to stored data.
3. Hash tables allow for immediate retrieval of values using a key, ideal for applications like databases and caching.
4. Hash tables can store different types of data, including strings, integers, and complex objects, making them versatile for various use cases.
5. Techniques like chaining or open addressing can resolve collisions effectively, ensuring that hash tables maintain fast operations even with high loads.

## **Disadvantages of Hash Tables**

1. If collisions become too frequent, it can degrade performance, especially if the hash table is not resized or managed properly.
2. In some cases, especially with large hash tables and a low load factor, the memory usage can be inefficient, leading to wasted space.
3. In rare cases (e.g., when a bad hash function is used), hash table operations can degrade to linear time (O(n)) if many collisions occur.
4. Dynamically resizing a hash table (rehashing) can be costly in terms of time and computational resources, especially with large datasets.
5. The performance of a hash table is heavily dependent on the quality of the hash function. A poorly designed hash function can lead to frequent collisions and inefficient data retrieval.
